Dans cet exercice, vous allez auditer le système d’information existant du Groupe Retail Sphère, analyser ses dépendances et évaluer les solutions technologiques disponibles afin de produire un audit structuré. L’audit comprendra :

une modélisation du système existant ;
une analyse fonctionnelle et technique ;
une étude des dépendances ;
un benchmark technologique argumenté.

Prenez connaissance du document de présentation du Groupe Retail Sphère afin d’identifier les activités principales, les utilisateurs et les objectifs du système d’information.

Votre objectif est de comprendre à quoi sert le système avant d’en analyser la structure technique.

En plus, listez les critères d’audit pour le préparer. Vous évaluerez l’adéquation du SI par rapport à ces critères.

 

Résultat attendu

Le périmètre fonctionnel du système est identifié.

Les principaux usages métier sont compris.

Les critères d’audit sont listés.

Recommandations

Identifiez les fonctionnalités principales (catalogue, commandes, stock, reporting).

Repérez les différents types d’utilisateurs.

Reformulez avec vos propres mots le rôle global du système.

Pour lister les critères de votre audit, inspirez-vous de la section ‘Rapport d’état du système d’information’ du document ‘Rapport métier haut niveau’. Par exemple, vous pouvez penser de : 

Performance

Maintenabilité

Cohérence des données

Architecture

Évolutivité

Sécurité

Vous pouvez partir du modèle de rapport d’audit du P5 de ce parcours.

Points de vigilance

Ne pas commencer directement par la technique.

Ne pas ajouter d’hypothèses non présentes dans les documents.

À partir du document technique fourni, identifiez les composants du système et représentez-les dans une modélisation simple.

Votre objectif est de produire une vue claire du système existant, qui servira de base à votre analyse.

 

Résultat attendu

Les composants principaux sont identifiés.

Une modélisation simple du système est réalisée.

Recommandations

Reprenez les composants décrits dans le document technique.

Représentez leurs relations principales.

Utilisez un schéma lisible (diagramme de composants ou schéma simple).

Points de vigilance

Ne pas inventer de nouveaux composants.

Ne pas surcharger le schéma.

Analysez chaque composant du système en précisant :

son rôle métier ;

sa responsabilité technique ;

ses interactions principales.

 

Résultat attendu

Chaque composant est décrit de manière structurée.

Les rôles et responsabilités sont clairement identifiés.

Recommandations

Utilisez une structure identique pour chaque composant.

Reliez chaque composant à un usage métier.

Appuyez-vous uniquement sur les informations fournies.

Points de vigilance

Ne pas faire une simple copie du document technique.

Ne pas rester trop général.

Étudiez les interactions entre les composants :

échanges de données ;

dépendances fonctionnelles ;

dépendances techniques.

Votre objectif est d’identifier les relations structurantes du système.

 

Résultat attendu

Les principales dépendances sont identifiées.

Leur impact est expliqué.

Recommandations

Concentrez-vous sur les flux principaux (commandes, stock, données).

Identifiez les composants fortement couplés.

Expliquez les conséquences de ces dépendances.

Points de vigilance

Ne pas lister des interactions sans analyse.

Ne pas traiter tous les flux au même niveau.

Ressources

Chapitres du cours OC ‘Assurez la qualité d'un projet SI avec ISO 9001’

À partir des technologies utilisées dans le système (Java, Node.js, PHP, etc.), identifiez des alternatives pertinentes.

Votre benchmark doit répondre à des problèmes observés dans le système.

 

Résultat attendu

Un benchmark structuré et pertinent est proposé.

Les solutions sont comparées selon des critères explicites.

Recommandations

Limitez-vous à quelques technologies pertinentes.

Définissez des critères de comparaison (maintenabilité, performance, cohérence).

Reliez chaque solution à un besoin identifié.

Points de vigilance

Ne pas faire une veille générale sans lien avec le système.

Ne pas multiplier les solutions inutilement.

Regroupez votre travail dans un document structuré.

Votre document doit permettre de comprendre :

le fonctionnement du système ;

ses limites principales (par rapport aux critères de l’audit);

les pistes d’amélioration possibles.

 

Résultat attendu

Le document d’audit structuré, cohérent et exploitable.

Recommandations

Structure type : contexte → criteres de l’audit -> système → dépendances → benchmark → synthèse.

Soignez la clarté et la lisibilité.

Vérifiez la cohérence globale.

Points de vigilance

Nous recommandons une longueur de 10 à 15 pages environ pour vous donner une indication du périmètre attendu.

Toutefois, aucune limite précise n'est imposée : l'essentiel est la pertinence de votre contenu plutôt que la quantité.
